1. デザイン原則（5原則）

"Good design is invisible. Merchants should feel like the tool is an extension of their intent — not a friction to navigate." — Polaris Design Principle, Shopify
1

Polaris-First

原則として Shopify Polaris の標準コンポーネントだけを使用する。独自スタイルは公開フォームのブランド表現のみに限定し、管理画面は 100% Polaris 準拠。色・余白・角丸・タイポは Polaris Design Tokens を直接参照。

2

Don't Make Me Think

各画面のプライマリアクションは 1 つだけ。CEO がドキュメント無しで初見で迷わず操作完了できることを QA 基準とする。EmptyState には次の行動を促す CTA を必ず配置。

3

Optimistic UI

承認 / 拒否 / トグル等のアクションは即座に UI 反映（楽観的更新）。サーバー失敗時はロールバック + Banner で通知。CEO が「待たされる感覚」を持たない速度を目指す。

4

State Completeness

全画面で「空 / 読込中 / エラー / 成功」の 4 状態を実装。skeleton ローディング、EmptyState illustration、Banner エラーは必須。デザインレビューは状態網羅マトリクスで合否判定。

5

Accessibility by Default

WCAG 2.1 AA 適合を最低ライン。Polaris 由来のキーボード操作・スクリーンリーダー対応は活用、独自要素のみ追加 ARIA を付与。Axe DevTools での自動監査を CI に組込み。

4. インフォメーションアーキテクチャ

5. ワイヤーフレーム（主要 8 画面）

W1: レビュー一覧（pending あり）

Daily Admin (CEO) が朝一番で見る画面。承認待ち N 件が即座に分かる。

ADMIN / TAB 1
UX 設計ノート
  • サイドバー: Shopify 標準サイドバー内に「📝 口コミ管理」が常駐。承認待ち件数を白いバッジで表示（一目で判断）。
  • KPI 行: 承認待ち件数を視覚的に最も目立たせる（橙色）。10 件超えは警告として知覚させる。
  • フィルタチップ: pending デフォルト選択。Polaris Filters component。1 クリックで切替。
  • テーブル行: 商品画像 + タイトル + 評価 + 本文 + 投稿者 + 種別 + 日付 + アクションを 1 行で完結。
  • アクション: 「承認」が primary (Polaris green), 「拒否」は secondary (赤)。同サイズ・隣接配置で誤クリック防止のため 8px gap 確保。
  • 一括承認: 右上に「★4以上を一括承認」便利機能（CEO の Daily 30 分目標達成のため）。
  • 種別バッジ: 認証購入（青）/ ギフト（ピンク）を視覚区別。誤情報誘発リスクの判定に使用。

W2: レビュー一覧（EmptyState）

初回起動時 / フィルタで該当 0 件の状態。次の行動を促す CTA を必ず配置。

EMPTY STATE
UX 設計ノート
  • Polaris EmptyState: イラスト + 1 行タイトル + 説明 + CTA ボタンの定番構造。
  • CTA: 「メール設定を開く」次の行動を 1 つだけ提示（複数の選択肢を出さない）。
  • 説明文: 「なぜ空なのか」「何をすれば解決するか」を端的に。

W3: レビュー詳細 Modal

テーブル行クリック時に展開。全情報を 1 画面で確認、承認 / 拒否を即決定。

MODAL
UX 設計ノート
  • Polaris Modal: 中央モーダル。背景は半透明黒で操作集中環境。
  • 1 画面完結: 商品・評価・タイトル・本文・写真・投稿者・種別・注文情報・自動ワードチェック結果を全て 1 画面で確認可能。
  • 自動ワードチェック: NG ワード辞書による事前判定（差別語等）。CEO の判断補助。
  • Footer Action: 「承認」が右端で primary 色、「拒否」が左隣で critical 色。誤クリック防止のため間隔確保。

W4: メール設定 - 新規作成フォーム

新 IP コラボのキャンペーン開始時の設定画面。10 分で完了することが目標。

ADMIN / TAB 2
UX 設計ノート
  • 3 ステップ構造: 「対象」→「日数」→「件名・本文」と上から順番に。マーケ担当が迷わない。
  • handle 入力時の自動検証: 「✓ ヒロアカ (23商品)」とリアルタイム表示。タイポを即時発見。
  • 変数ヒント: 本文ラベルに変数一覧を併記。`{{customer_name}}` `{{product_title}}` `{{review_url}}` が見たまま使える。
  • ライブプレビュー: 右側に常時プレビュー表示。実データ代入後の最終形を確認しながら編集。
  • 作成ボタン文言: 「作成（無効状態）」と明示。誤って全件送信を防止（opt-in 明確化）。

W5: お客様向け投稿フォーム（モバイル）

メール内の URL から開く投稿画面。3 分以内に完了する設計。

CUSTOMER / MOBILE
UX 設計ノート
  • モバイルファースト: 75% のお客様がスマホで開くと想定。デザインはモバイル基準でスタート。
  • ASTROMEDA ブランド統一: 管理画面（Polaris）と異なり、お客様向けは ASTROMEDA ダークテーマで統一感。
  • 3 分完了の徹底: 入力欄を最小限。星評価 / 見出し / 本文 / 写真 / 表示名 / 規約同意 / 投稿 の 7 ステップ。
  • 「本名でなくて可」: ペルソナ#1 田中翔太の最大の懸念を即座に解消。投稿率を 15% 向上見込み。
  • autosave: localStorage で入力中の内容を保存。誤って閉じても復元可能。
  • 競合との差別化: Yotpo / Judge.me より速い・写真多い・匿名性高い。

削除すると、この設定で予約された送信予定が全てキャンセルされます。この操作は取り消せません。

有効化すると、対象注文の発送 {N} 日後に自動メール送信が開始されます。

9. アクセシビリティ仕様（WCAG 2.1 AA）

キーボード操作

  • 全インタラクティブ要素を Tab キーで到達可能
  • 承認 / 拒否ボタンに Enter / Space で実行
  • Modal は ESC キーで閉じる
  • テーブル行を矢印キーで移動（Polaris IndexTable 標準）
  • Tab タブを矢印キーで切替（Polaris Tabs 標準）
  • Skip to main content リンク（先頭）

スクリーンリーダー対応

  • aria-label / aria-labelledby 適切に設定
  • aria-live 領域で Toast / Banner を announce
  • aria-busy で読込中状態を伝達
  • 承認状態変化を aria-live で読み上げ
  • Modal は aria-modal="true" + focus trap
  • 装飾アイコンには aria-hidden="true"

視覚的アクセシビリティ

  • 色のコントラスト: テキスト 4.5:1 以上 / UI 要素 3:1 以上
  • 色だけで情報を伝えない（status はバッジ + アイコン）
  • focus indicator が明確（Polaris 標準 outline）
  • テキストは 200% zoom でも崩れない
  • クリック可能領域 44×44px 以上

動き・モーション

  • prefers-reduced-motion 尊重
  • 自動再生・自動更新は実装しない（手動 reload）
  • アニメーションは 200ms 以内・linear / ease-out
  • Polaris motion tokens 使用
